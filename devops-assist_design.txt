  시스템 가치 평가

  이 아이디어가 의미있는 이유

  1. 실제 pain point 존재
  - 15개 프로젝트를 찾아다니며 파악해야 함, 통합 검색 없음
  - Hive_v2 패턴이 여러 프로젝트에서 참조되지만 연결고리가 없음
  - 같은 기술(SSE, WebSocket, Gemini)이 반복 구현되는데 재사용 지식이 없음

  2. 기존 프로젝트와 시너지
  - polymath — 키워드→지식 그래프 이미 구현됨 → 도메인만 "프로젝트 문서"로
  바꾸면 됨
  - dev-agent — AI 코드 분석 파이프라인 존재
  - Hive_v2 — 멀티에이전트 패턴 재사용 가능

  3. 포트폴리오 차별성
  - "15개 프로젝트를 AI로 자동 분류/연결/검색하는 시스템"은 강력한 포트폴리오
  아이템

  주의할 점

  polymath와 중복 위험
  - polymath가 이미 "키워드→지식 그래프"를 한다
  - 새 시스템이 polymath의 확장인지, 별개 시스템인지 명확히 해야 함

  범위 폭발 위험
  - "문서 관리"는 너무 넓음 — Notion/Confluence와 경쟁하게 됨
  - 이 워크스페이스 전용으로 범위를 좁혀야 함

  ---
  추천 방향: 범위를 좁혀라

  "범용 문서관리 시스템"이 아니라 "Project Intelligence Hub"

  핵심 기능 3가지만:

  1. 프로젝트 자동 인덱싱
     - C:/dep 하위 모든 README, CLAUDE.md, *.txt, *.md 파일 자동 감지
     - 파일 변경 감지 → 재인덱싱

  2. AI 기반 통합 검색
     - "JWT 어떻게 구현했어?" → dev-agent, TableQ, Messenger에서 관련 코드/문서
  검색
     - Gemini로 답변 생성

  3. 크로스-프로젝트 지식 그래프
     - polymath 방식으로 프로젝트 간 공통 기술/패턴 시각화
     - "Gemini 사용 프로젝트 3개", "WebSocket 사용 4개" 등 자동 연결

  ---
  기술 스택 제안

  Backend  : Spring Boot 3.4.2 (Maven) — 기존 패턴 그대로
  AI       : Gemini 2.5-flash — 기존 GeminiService 패턴 재사용
  Agent    : Hive_v2 A~I 패턴 → IndexerAgent + SearchAgent + GraphAgent
  File     : Java NIO WatchService (파일 변경 감시)
  Vector   : 간단히 H2 + 텍스트 유사도 (pgvector로 업그레이드 여지)
  Frontend : Thymeleaf + D3.js — polymath UI 재사용
  Port     : 8082
